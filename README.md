# 🏰 CastleKeeper

**Telegram bot — castle keeper**  
Ask it about a castle, and it will answer like a real gatekeeper!


## ⚔️ About

CastleKeeper is an ecosystem for managing a database of medieval castles through Telegram.  
You message the bot, and it returns a list of strongholds from a real PostgreSQL database.  
Add a new castle via REST API — and the bot is ready to tell its story right away.

The project was born from a desire to learn Spring Boot, Docker, and clean architecture — all wrapped in a castle theme.


## 🏗 Architecture

| Component          | Technology                     | Description                                 |
| ------------------ | ------------------------------ | ------------------------------------------- |
| **Telegram Bot**   | Java 25 (LTS), Maven           | Processes commands, calls REST API          |
| **REST API**       | Spring Boot 3, Java 17, Flyway | CRUD for castles, authors, types, materials |
| **Database**       | PostgreSQL 16 (Docker)         | Stores all entities with FK relationships   |
| **Documentation**  | Obsidian (root vault)          | Notes, architecture, API contracts, canvas  |
| **Infrastructure** | Docker Compose                 | One-command PostgreSQL setup                |

### Data flow

```
Telegram (user) → Bot (Java 25) → HTTP → Server (Spring Boot 3, Java 17) → PostgreSQL
```

> 💡 Why different Java versions? The bot uses JDK 25 as LTS, the server uses Java 17 for maximum compatibility with Spring Boot 3.x.

## 🗄 Database Schema

5 entities with FK relationships — schema managed by **Flyway migrations** in `server/src/main/resources/db/migration/`.

```
author_type ──< author ──< castle >── material
                 │            │
                 └──< reconstruction
```

| Table | Description | Key Columns |
|-------|-------------|-------------|
| `author_type` | Author category (builder, chronicler, etc.) | `id` PK, `name` UNIQUE, `description` |
| `author` | Castle author / builder | `id` PK, `name`, `author_type_id` FK |
| `material` | Building material | `id` PK, `name` UNIQUE |
| `castle` | The castle itself | `id` PK, `name`, `description`, `author_id` FK, `built_year`, `destroyed_year`, `height_m`, `material_id` FK |
| `reconstruction` | Reconstruction event | `id` PK, `castle_id` FK, `author_id` FK, `reconstruction_year` |

> 📖 Full schema docs: [`docs/database-schema/`](docs/database-schema/README.md) — auto-generated from the live database via [`tbls`](https://github.com/k1LoW/tbls) in CI.


## 🤖 Bot Commands

The bot responds with a medieval narrative style. Selection-based actions use **inline keyboards** (5 items per page, ◀ ▶ pagination) — no manual ID entry.

### 🏰 Castles
| Command / Action | Description |
|------------------|-------------|
| `/castles` | List all castles (text output) |
| `/castlemenu` → 👁️ Select | View castle via inline keyboard picker |
| `/random` | Random castle (text output) |
| `/addcastle` | 7-step wizard: name → desc → author → years → height → material → confirm |
| `/castlemenu` → ✒️ Edit | Pick castle → pick field → enter value / pick FK → confirm |
| `/castlemenu` → 💥 Delete | Pick castle → confirm ✅/❌ |

## 🔌 API Endpoints

**Base URL:** `http://localhost:${SERVER_PORT:8080}/api/` (local) · `https://${DOMAIN}/api` (production)

> 📄 Machine-readable specification: [`server/src/main/resources/api-docs.json`](server/src/main/resources/api-docs.json) (OpenAPI 3.0)  
> 🖥️ Interactive explorer: `http://localhost:${SERVER_PORT:8080}/swagger-ui.html`
>
> 💡 **DTO Pattern:** All endpoints use dedicated Request/Response DTOs. POST/PUT accept FK fields as plain IDs (`authorTypeId`, `materialId`). GET returns nested objects with full display data. See [`docs/architecture.md`](docs/architecture.md) for details.

### 🏰 Castles

|--------|----------|-------------|--------|
| GET | `/api/castles` | List all castles | 200 |
